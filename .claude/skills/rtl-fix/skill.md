# RTL Fix Skill

Comprehensive detection and fixing of Right-to-Left (RTL) issues in Hebrew web interfaces. Covers CSS, HTML, JavaScript, React/Next.js, and Tailwind CSS v4.

## Instructions

When invoked, scan the codebase for RTL issues and fix them. Work through each category below systematically.

---

## CATEGORY 1: Physical CSS Properties (Must Replace with Logical)

Physical properties don't adapt to text direction. Replace with logical equivalents.

### Detection Patterns (Tailwind classes)

| Physical (WRONG in RTL) | Logical (CORRECT) | Notes |
|---|---|---|
| `ml-*` | `ms-*` | margin-inline-start |
| `mr-*` | `me-*` | margin-inline-end |
| `pl-*` | `ps-*` | padding-inline-start |
| `pr-*` | `pe-*` | padding-inline-end |
| `left-*` | `start-*` | inset-inline-start |
| `right-*` | `end-*` | inset-inline-end |
| `text-left` | `text-start` | text-align |
| `text-right` | `text-end` | text-align |
| `float-left` | `float-start` | float |
| `float-right` | `float-end` | float |
| `border-l-*` | `border-s-*` | border-inline-start |
| `border-r-*` | `border-e-*` | border-inline-end |
| `rounded-l-*` | `rounded-s-*` | border-start-radius |
| `rounded-r-*` | `rounded-e-*` | border-end-radius |
| `rounded-tl-*` | `rounded-ss-*` | start-start corner |
| `rounded-tr-*` | `rounded-se-*` | start-end corner |
| `rounded-bl-*` | `rounded-es-*` | end-start corner |
| `rounded-br-*` | `rounded-ee-*` | end-end corner |
| `scroll-ml-*` | `scroll-ms-*` | scroll-margin-start |
| `scroll-mr-*` | `scroll-me-*` | scroll-margin-end |
| `scroll-pl-*` | `scroll-ps-*` | scroll-padding-start |
| `scroll-pr-*` | `scroll-pe-*` | scroll-padding-end |

### Detection Patterns (Raw CSS)

| Physical (WRONG) | Logical (CORRECT) |
|---|---|
| `margin-left` | `margin-inline-start` |
| `margin-right` | `margin-inline-end` |
| `padding-left` | `padding-inline-start` |
| `padding-right` | `padding-inline-end` |
| `border-left` | `border-inline-start` |
| `border-right` | `border-inline-end` |
| `left:` (positioning) | `inset-inline-start` |
| `right:` (positioning) | `inset-inline-end` |
| `text-align: left` | `text-align: start` |
| `text-align: right` | `text-align: end` |
| `float: left` | `float: inline-start` |
| `float: right` | `float: inline-end` |
| `border-top-left-radius` | `border-start-start-radius` |
| `border-top-right-radius` | `border-start-end-radius` |
| `border-bottom-left-radius` | `border-end-start-radius` |
| `border-bottom-right-radius` | `border-end-end-radius` |

### EXCEPTIONS -- Do NOT Replace

- **Intentional LTR blocks**: Elements with `ltr` class or `dir="ltr"` -- numbers, currency, percentages are often intentionally LTR. In this codebase, `ltr text-right` is a common pattern for displaying numbers right-aligned in LTR direction. Leave these alone.
- **Dialog/modal centering**: `left-1/2 -translate-x-1/2` for centering is direction-independent. Leave alone.
- **Chart library margins**: Recharts/chart library config objects like `margin={{ left: 0, right: 0 }}` are not CSS -- leave alone.
- **Toggle/switch positioning**: `left-[3px] right-auto` patterns for toggle knob position are physical by design. Leave alone.
- **Progress bar absolute positioning**: `left: X%` with `style=` for progress bar fills -- these are math-based, not directional.
- **shadcn/ui components**: Files in `src/components/ui/` that are generated by shadcn -- only fix if clearly broken, as these have their own conventions.
- **Fixed/absolute sidebar/drawer positioning**: `right-0` on fixed sidebars -- see Tailwind v4 known issues below.

### TAILWIND V4 LOGICAL PROPERTIES -- KNOWN ISSUES

**CRITICAL**: Some Tailwind v4 logical property classes do NOT work reliably in all contexts. Discovered in production:

| Class | Status | Workaround |
|---|---|---|
| `end-0` / `start-0` (positioning) | UNRELIABLE -- broke sidebar positioning when replacing `right-0` | Use physical `right-0` / `left-0` with comment |
| `border-s-*` / `border-e-*` (border width) | WORKS in v4 -- tested and confirmed | Use freely |
| `ms-*` / `me-*` (margin) | WORKS | Use freely |
| `ps-*` / `pe-*` (padding) | WORKS | Use freely |
| `text-start` / `text-end` | WORKS | Use freely |
| `rounded-ss-*` etc. (corners) | WORKS but verify visually | Test after changing |
| `inset-s-*` / `inset-e-*` | Same as start/end -- may have issues | Test carefully |

**Rule**: When replacing physical positioning classes (`right-0`, `left-0`, `right-[value]`, `left-[value]`) with logical equivalents (`end-0`, `start-0`), ALWAYS test visually. If the element is `position: fixed` or `position: absolute` with a sidebar/drawer/panel, prefer physical properties + a `/* RTL: physical intentional */` comment.

## CATEGORY 2: Hebrew Currency (Shekel) and Number Formatting

### Shekel Symbol Position

The Academy of the Hebrew Language recommends placing the shekel sign to the LEFT of the number. However, in actual Israeli digital usage and in `Intl.NumberFormat('he-IL')`, the symbol appears AFTER the number with proper bidi marks.

**What `Intl.NumberFormat('he-IL', { style: 'currency', currency: 'ILS' })` actually produces:**

```
Positive:  RLM + "1,234" + NBSP + RLM + "₪"    → visually: ₪ 1,234
Negative:  RLM + LRM + "-500" + NBSP + RLM + "₪" → visually: ₪ 500-
Range:     RLM + "500-1,000" + NBSP + RLM + "₪"  → single ₪ at end
```

Character breakdown (from `formatToParts`):
- Starts with RLM (U+200F) to establish RTL context
- Negative numbers get LRM (U+200E) before minus to keep it positioned correctly
- Currency symbol comes AFTER the number, separated by non-breaking space (U+00A0) + RLM
- The Intl API handles all bidi marks automatically

**Practical rule for this codebase:**
```tsx
// CORRECT -- use Intl.NumberFormat, it handles everything
export function formatCurrency(amount: number): string {
  return new Intl.NumberFormat('he-IL', {
    style: 'currency',
    currency: 'ILS',
    maximumFractionDigits: 0,
  }).format(amount)
}

// Display with dir="ltr" to isolate from surrounding RTL text
<span dir="ltr">{formatCurrency(amount)}</span>
// or
<bdi>{formatCurrency(amount)}</bdi>
```

**NEVER do this:**
```tsx
// WRONG -- manual shekel placement will fight the bidi algorithm
<span>₪{amount}</span>
<span>{amount} ₪</span>
<span>₪ {amount.toLocaleString()}</span>
```

### Negative Currency Display

`Intl.NumberFormat('he-IL')` formats negatives as `-500 ₪` with embedded LRM/RLM marks. The minus sign stays on the correct side automatically. No manual intervention needed.

For accounting format (parentheses instead of minus):
```tsx
new Intl.NumberFormat('he-IL', {
  style: 'currency',
  currency: 'ILS',
  currencySign: 'accounting'
})
```

### Currency Ranges

`Intl.NumberFormat` supports `formatRange()`:
```tsx
const fmt = new Intl.NumberFormat('he-IL', { style: 'currency', currency: 'ILS' });
fmt.formatRange(500, 1000); // "500-1,000 ₪" (single ₪ at end, with bidi marks)
```

### Percentage Display

In Hebrew, percentage sign goes AFTER the number, same as English: **50%** (not %50).

```tsx
// Both correct:
new Intl.NumberFormat('he-IL', { style: 'percent' }).format(0.5) // "50%"
`${Math.round(value * 100)}%`  // "50%"
```

**Warning**: Some RTL layout engines may push % to the left. If you see `%50` in your UI, wrap in `<span dir="ltr">50%</span>`.

## CATEGORY 3: Punctuation and Symbols in Hebrew RTL

### Exclamation and Question Marks

Punctuation goes at the END of the sentence. In Hebrew (RTL), the end is on the LEFT side. The browser handles this automatically through the bidi algorithm.

```
Hebrew:  !שלום    ← The ! appears on the left (end of sentence)
English: Hello!   ← The ! appears on the right (end of sentence)
```

These are "neutral" characters in Unicode bidi -- they inherit direction from surrounding text. **No special handling needed** for pure Hebrew text.

**Problem case**: Mixed Hebrew + numbers/English at sentence boundaries:
```tsx
// MAY break -- punctuation can jump to wrong side
<span>נותרו {daysLeft} ימים!</span>

// SAFE -- isolate the number
<span>נותרו <bdi>{daysLeft}</bdi> ימים!</span>
```

### Parentheses

Parentheses are "mirrored" characters in Unicode. In RTL context, `(` visually becomes `)` and vice versa. This is AUTOMATIC and CORRECT.

```
Code:    (שלום)
Display: (שלום)  ← the opening paren is on the right in RTL
```

**Problem case**: Mixed content:
```tsx
// MAY break
<span>הוצאה (Expense)</span>

// SAFE
<span>הוצאה <bdi>(Expense)</bdi></span>
```

### Quotation Marks in Hebrew

Modern Hebrew uses the same quotation marks as English: "text" (both marks high). This is standard for web UIs.

The gershayim (U+05F4) is a Hebrew-specific mark used for abbreviations (צה"ל) and Hebrew numerals (י"ח for 18). Most people substitute regular double-quotes since gershayim isn't on standard keyboards.

### Colons and Semicolons

Colons and semicolons are neutral characters. They inherit direction from context.

**Critical issue**: A colon between Hebrew text and a number can "jump":
```tsx
// RISKY -- colon may appear after the number
<span>סה"כ: {total}</span>

// SAFE -- isolate the number
<span>סה"כ: <bdi>{formatCurrency(total)}</bdi></span>
```

### How Neutral Characters Behave

The bidi algorithm resolves neutral characters (spaces, punctuation) by looking at surrounding strong characters. A neutral character between two characters of the SAME direction assumes that direction. Between characters of DIFFERENT directions, the neutral takes the direction of the paragraph (RTL in Hebrew).

This means:
- `"john:"` followed by Hebrew displays colon on the left of "john"
- `"john:"` followed by English displays colon on the right of "john"

**Rule**: When mixing directions near punctuation, use `<bdi>` to isolate.

### Plus/Minus Signs with Numbers

Plus (+) and minus (-) are "weak" bidi characters. They follow adjacent characters.

```tsx
// In RTL context, minus may detach from number
// WRONG
<span>-500</span>

// CORRECT
<span dir="ltr">-500</span>
// Or use Intl formatting which adds proper bidi marks
```

## CATEGORY 4: Numbers in Hebrew RTL Context

### Core Rule

Numbers (digits 0-9) are ALWAYS displayed left-to-right, even inside RTL text. The Unicode bidi algorithm handles this automatically. Digits are classified as "weak LTR" -- they form LTR runs within RTL text.

### Phone Numbers

Phone numbers with dashes can break across bidi boundaries:
```tsx
// MAY render incorrectly in RTL context
<span>054-123-4567</span>

// SAFE
<span dir="ltr">054-123-4567</span>
// or
<a href="tel:+972541234567" dir="ltr">054-123-4567</a>
```

### Dates

Date strings are LTR sequences (digits + separators):
```tsx
// Usually works fine
<span>19/03/2026</span>

// But if preceded by Hebrew text, isolate:
<span>תאריך: <bdi>19/03/2026</bdi></span>
```

### Credit Card / ID Numbers

Always force LTR:
```tsx
<span dir="ltr">1234 5678 9012 3456</span>
<span dir="ltr">012345678</span>
```

### Numbers Inside Hebrew Sentences

When a number is embedded in Hebrew text, the bidi algorithm usually handles it. Problems arise at BOUNDARIES:

```tsx
// Usually fine -- number surrounded by Hebrew on both sides
<p>קניתי 3 פריטים</p>

// Problematic -- number at end with punctuation
<p>סה"כ: 1,500</p>

// Safe
<p>סה"כ: <bdi>1,500</bdi></p>
```

**Rule of thumb**: If a number is surrounded by Hebrew on both sides, it's fine. If it's at a text boundary or adjacent to punctuation, wrap in `<bdi>` or `<span dir="ltr">`.

## CATEGORY 5: Headings and Text Alignment

### Default Alignment

All text in Hebrew RTL should be RIGHT-aligned by default. Setting `dir="rtl"` on `<html>` makes `text-align: start` = right-alignment.

### Headings

- **Page titles**: Right-aligned (default in RTL). Standard for Hebrew dashboard/app UI.
- **Section headings**: Right-aligned (default).
- **Centered headings**: Acceptable ONLY for hero sections, marketing pages, empty states, call-to-action blocks. NOT for dashboard/app headings.

```tsx
// Default -- right-aligned (correct for app headings)
<h1 className="text-2xl font-bold">סיכום חודשי</h1>

// Centered -- only for hero/empty state
<h1 className="text-2xl font-bold text-center">ברוכים הבאים</h1>
```

### Sub-headings

Follow same alignment as parent heading. Right-aligned by default.

## CATEGORY 6: Directional Icons and UI Elements

### Icons That MUST Flip

- Arrow left/right (navigation: back, forward, next, previous)
- Chevron left/right (breadcrumbs, carousels, pagination)
- External link arrow
- Reply / forward icons
- Undo / redo icons
- Indent / outdent icons
- List bullet alignment indicators
- Logout/login door icons with arrows
- Text alignment icons
- Sort indicators with horizontal arrows

### Icons That Must NOT Flip

- Search (magnifying glass)
- Checkmarks
- X / close
- Plus / minus
- Media controls (play, pause, stop) -- tape direction, not text direction
- Warning triangles
- Settings / gear
- User / avatar
- Download / upload arrows (vertical)
- Refresh / sync (circular arrows -- clock direction is universal)
- Home, star, heart, flag, phone, clock, calendar, lock, eye, trash

### How to Flip

```tsx
// Method 1: Tailwind rtl variant (PREFERRED)
<ChevronRight className="rtl:-scale-x-100" />

// Method 2: CSS custom property
[dir="rtl"] { --icon-dir: -1; }
[dir="ltr"] { --icon-dir: 1; }
<Icon style={{ transform: `scaleX(var(--icon-dir))` }} />

// Method 3: Swap icon entirely
{isRTL ? <ChevronLeft /> : <ChevronRight />}
```

### Progress Bars

Progress bars should fill RIGHT-TO-LEFT in Hebrew RTL, matching reading direction.

**Exception**: Media playback progress bars fill left-to-right (tape/time direction).

```tsx
// Budget progress bar -- RTL fill (use dir="rtl" on container)
<div className="w-full bg-gray-800 rounded-full h-2" dir="rtl">
  <div className="h-2 rounded-full" style={{ width: `${percent}%` }} />
</div>
```

### Sliders / Range Inputs

In RTL, minimum on RIGHT, maximum on LEFT. Native `<input type="range">` respects `dir="rtl"`.

```tsx
<input type="range" dir="rtl" min={0} max={100} />
```

### Detection

Search for: `ChevronRight`, `ChevronLeft`, `ArrowRight`, `ArrowLeft`, `ArrowUpRight`, `ExternalLink`, `Reply`, `Forward`, `Undo`, `Redo` from lucide-react. Check if they have RTL flip handling.

## CATEGORY 7: Forms in RTL

### Labels

Labels above inputs (standard). Both are right-aligned by default in RTL.

```tsx
<div>
  <label className="block text-sm mb-1">שם</label>
  <input className="w-full" />
</div>
```

### Placeholder Text Direction

```tsx
// Hebrew placeholder -- fine as-is
<input placeholder="הכנס שם..." />

// Number placeholder -- use dir="auto"
<input placeholder="0.00" dir="auto" />

// Mixed content input (search)
<input type="search" dir="auto" placeholder="חיפוש..." />
```

### Validation Messages

Right-aligned, below input (automatic in RTL context):
```tsx
{error && <p className="text-red-500 text-sm mt-1">{error}</p>}
```

### Mixed Content Inputs

```tsx
// dir="auto" detects first strong character
<input dir="auto" />

// Number-only inputs -- force LTR, align to end
<input type="number" dir="ltr" className="text-end" />
```

### Dropdown Menus

In RTL, dropdowns open aligned to the RIGHT edge (start side). Most component libraries handle this with `dir="rtl"`.

**Select arrow**: Should appear on the LEFT side in RTL. The `tailwindcss-forms` plugin puts it on the right -- may need override:
```css
[dir="rtl"] select {
  background-position: left 0.5rem center;
  padding-right: 0.75rem;
  padding-left: 2.5rem;
}
```

### Email / URL / Phone Inputs

Always LTR:
```css
input[type="email"],
input[type="url"],
input[type="tel"] {
  direction: ltr;
  text-align: right;
}
```

## CATEGORY 8: Tables in RTL

### Column Order

Tables auto-reverse column order in RTL. First `<th>`/`<td>` in HTML appears on the RIGHT. This is correct. Do NOT manually reverse HTML column order.

### Number Columns

Numbers display LTR within RTL table automatically. For formatted currency with bidi marks, use `dir="ltr"` on the cell:
```tsx
<td dir="ltr" className="text-end">{formatCurrency(1500)}</td>
```

### Sort Indicators

Up/down arrows stay the same. Any horizontal sort arrows should flip. Most table libraries handle this.

### Table Alignment

- Text columns: right-aligned (default in RTL)
- Number columns: left-aligned (end of row in RTL)
- Action columns: left side of table (last visual column in RTL)

## CATEGORY 9: Box Shadow and Gradients

### Box Shadow

Horizontal offset does NOT auto-flip. Use CSS custom property:
```css
[dir="ltr"] { --shadow-dir: 1; }
[dir="rtl"] { --shadow-dir: -1; }
.card { box-shadow: calc(4px * var(--shadow-dir)) 4px 8px rgba(0,0,0,0.2); }
```

### Background Gradients

`linear-gradient(to right, ...)` does NOT auto-flip:
```tsx
className="bg-gradient-to-r rtl:bg-gradient-to-l"
```

## CATEGORY 10: Transforms and Animations

### translateX Does NOT Auto-Flip

```css
[dir="ltr"] { --dir: 1; }
[dir="rtl"] { --dir: -1; }
.slide-in { transform: translateX(calc(-100% * var(--dir))); }
```

Tailwind:
```tsx
className="translate-x-full rtl:-translate-x-full"
```

### CSS Animations with Directional Movement

`@keyframes` with `translateX` or horizontal movement need RTL variants.

## CATEGORY 11: Tailwind CSS v4 RTL Specifics

### `dir="rtl"` vs `direction: rtl` vs Tailwind

**Always use the HTML attribute `dir="rtl"`** on the `<html>` element. W3C recommendation.

Why:
1. CSS `direction: rtl` is NOT picked up by `:dir(rtl)` pseudo-class or Tailwind's `rtl:` variant
2. Works even if CSS fails to load
3. CSS `direction` doesn't inherit into table columns properly
4. The `dir` attribute influences the bidi algorithm at the HTML level

```html
<!-- CORRECT -->
<html lang="he" dir="rtl">

<!-- WRONG -->
<html lang="he" style="direction: rtl">
```

### The `rtl:` Variant -- When to Use

Use `rtl:` ONLY when no logical property equivalent exists or works:

```tsx
// Gradient direction (no logical equivalent)
className="bg-gradient-to-r rtl:bg-gradient-to-l"

// Icon flipping
className="rtl:-scale-x-100"

// translateX animations
className="translate-x-full rtl:-translate-x-full"

// space-x (prefer gap-* instead)
className="flex space-x-4 rtl:space-x-reverse"
```

**Do NOT use `rtl:` when logical property works:**
```tsx
// WRONG
className="ml-4 rtl:ml-0 rtl:mr-4"

// CORRECT
className="ms-4"
```

### Logical Properties -- Working vs Broken in v4

**Confirmed working:**
- `ms-*` / `me-*` (margin start/end)
- `ps-*` / `pe-*` (padding start/end)
- `text-start` / `text-end`
- `border-s-*` / `border-e-*` (border start/end width)
- `rounded-ss-*` / `rounded-se-*` / `rounded-es-*` / `rounded-ee-*`
- `float-start` / `float-end`
- `gap-*` (direction-independent)

**Known issues:**
- `start-0` / `end-0` (positioning) -- BROKE a fixed sidebar in production. Use physical `right-0`/`left-0` for fixed/absolute panels.
- `inset-s-*` / `inset-e-*` -- same issue as start/end

### The `.ltr` Class Pattern

This codebase defines `.ltr { direction: ltr; }` in globals.css. The pattern `ltr text-right` forces LTR for numbers while right-aligning them in the RTL layout. This is intentional -- do not remove.

## CATEGORY 12: Flexbox and Grid

### Flexbox

Flexbox `row` auto-reverses in RTL. No action needed for basic layouts.

`space-x-*` uses `margin-left` -- needs `space-x-reverse` or replace with `gap-*`:
```tsx
// Best approach
<div className="flex gap-4">
```

### Grid

CSS Grid auto-reverses column flow in RTL. No special handling needed.

### Table Column Order

Tables auto-reverse in RTL. First HTML column = rightmost visual column. Correct.

## CATEGORY 13: Scrollbar and Overflow

In RTL:
- Vertical scrollbar on the LEFT
- Horizontal overflow goes LEFT
- Firefox has historical bugs with horizontal scrollbars in RTL
- Prevent phantom horizontal scroll: `html[dir="rtl"] { overflow-x: hidden; }`

## CATEGORY 15: Common Mistakes Checklist

1. **Using `text-right` instead of `text-start`**: In RTL, right IS start. `text-right` works visually but `text-start` is semantically correct and future-proof.

2. **Forgetting to isolate numbers at text boundaries**: `סה"כ: 1,500` -- colon may jump. Always wrap boundary numbers in `<bdi>`.

3. **Flipping icons that shouldn't flip**: Play buttons, checkmarks, clocks are universal.

4. **NOT flipping icons that should flip**: Back arrows, chevrons, external link indicators.

5. **Manual currency formatting**: Using `₪${amount}` instead of `Intl.NumberFormat`. The Intl API adds proper bidi marks.

6. **Assuming `start-0`/`end-0` work everywhere**: Logical positioning breaks with `position: fixed` in Tailwind v4.

7. **Using physical margin/padding in CSS**: Use `margin-inline-start`/`padding-inline-end`.

8. **Forgetting `space-x-reverse`**: Use `gap-*` instead.

9. **Assuming translateX auto-flips**: It doesn't. Slide animations need explicit RTL handling.

10. **Not setting `dir="ltr"` on email/tel/url inputs**: These must always be LTR.

### Browser-Specific RTL Issues

**Chrome (Blink)**: Best RTL support overall. May have edge cases with `position: fixed` + logical inset.

**Firefox (Gecko)**: Historical horizontal scrollbar bugs in RTL. `scrollLeft` returns NEGATIVE values in RTL (opposite of Chrome).

**Safari (WebKit)**: All iOS browsers use WebKit. `position: sticky` can have RTL issues. Auto-zoom on input focus disrupts layout -- prevent with `font-size: 16px` on inputs. `scrollLeft` starts at max value, decreases to 0 (different from both Chrome and Firefox).

**scrollLeft differences in RTL:**
```
Chrome:  starts at 0 (rightmost), positive = scroll left
Firefox: starts at 0 (rightmost), NEGATIVE = scroll left
Safari:  starts at max (rightmost), decreases toward 0
```
Use `element.scrollIntoView()` or `scroll-snap` instead of manual `scrollLeft`.

## SCANNING PROCEDURE

When running this skill:

1. **Scan globals.css** for physical properties in CSS rules
2. **Scan all .tsx files** for physical Tailwind classes (ml-, mr-, pl-, pr-, left-, right-, text-left, text-right, border-l-, border-r-, rounded-l-, rounded-r-)
3. **For each match**, check exceptions (ltr block, chart config, shadcn, centering, toggle, fixed positioning)
4. **Check icon usage** -- directional Lucide icons need RTL flip
5. **Check form inputs** -- email/url/tel/number need proper direction
6. **Check currency/number displays** -- verify `Intl.NumberFormat('he-IL')` and `<bdi>`/`dir="ltr"` wrappers
7. **Check `space-x-*`** -- suggest `gap-*`
8. **Check transforms/animations** -- translateX needs RTL handling
9. **Check progress bars/sliders** -- should fill RTL direction
10. **Report findings** structured with file:line and fix

## OUTPUT FORMAT

```
FILE: src/app/example/page.tsx:42
ISSUE: Physical class `mr-2` used instead of logical `me-2`
CONTEXT: <span className="mr-2 text-sm">
FIX: Replace `mr-2` with `me-2`
SEVERITY: low|medium|high
```

Severity guide:
- **high**: Layout visually broken in RTL
- **medium**: Inconsistent with RTL best practices, may cause subtle issues
- **low**: Physical property in a context where it doesn't matter
