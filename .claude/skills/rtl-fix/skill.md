# RTL Fix Skill

Comprehensive detection and fixing of Right-to-Left (RTL) issues in Hebrew web interfaces. Covers CSS, HTML, JavaScript, React/Next.js, and Tailwind CSS.

## Instructions

When invoked, scan the codebase for RTL issues and fix them. Work through each category below systematically.

---

## CATEGORY 1: Physical CSS Properties (Must Replace with Logical)

Physical properties don't adapt to text direction. Replace with logical equivalents.

### Detection Patterns (Tailwind classes)

| Physical (WRONG in RTL) | Logical (CORRECT) | Notes |
|---|---|---|
| `ml-*` | `ms-*` | margin-inline-start |
| `mr-*` | `me-*` | margin-inline-end |
| `pl-*` | `ps-*` | padding-inline-start |
| `pr-*` | `pe-*` | padding-inline-end |
| `left-*` | `start-*` | inset-inline-start |
| `right-*` | `end-*` | inset-inline-end |
| `text-left` | `text-start` | text-align |
| `text-right` | `text-end` | text-align |
| `float-left` | `float-start` | float |
| `float-right` | `float-end` | float |
| `border-l-*` | `border-s-*` | border-inline-start |
| `border-r-*` | `border-e-*` | border-inline-end |
| `rounded-l-*` | `rounded-s-*` | border-start-radius |
| `rounded-r-*` | `rounded-e-*` | border-end-radius |
| `rounded-tl-*` | `rounded-ss-*` | start-start corner |
| `rounded-tr-*` | `rounded-se-*` | start-end corner |
| `rounded-bl-*` | `rounded-es-*` | end-start corner |
| `rounded-br-*` | `rounded-ee-*` | end-end corner |
| `scroll-ml-*` | `scroll-ms-*` | scroll-margin-start |
| `scroll-mr-*` | `scroll-me-*` | scroll-margin-end |
| `scroll-pl-*` | `scroll-ps-*` | scroll-padding-start |
| `scroll-pr-*` | `scroll-pe-*` | scroll-padding-end |

### Detection Patterns (Raw CSS)

| Physical (WRONG) | Logical (CORRECT) |
|---|---|
| `margin-left` | `margin-inline-start` |
| `margin-right` | `margin-inline-end` |
| `padding-left` | `padding-inline-start` |
| `padding-right` | `padding-inline-end` |
| `border-left` | `border-inline-start` |
| `border-right` | `border-inline-end` |
| `left:` (positioning) | `inset-inline-start` |
| `right:` (positioning) | `inset-inline-end` |
| `text-align: left` | `text-align: start` |
| `text-align: right` | `text-align: end` |
| `float: left` | `float: inline-start` |
| `float: right` | `float: inline-end` |
| `border-top-left-radius` | `border-start-start-radius` |
| `border-top-right-radius` | `border-start-end-radius` |
| `border-bottom-left-radius` | `border-end-start-radius` |
| `border-bottom-right-radius` | `border-end-end-radius` |

### EXCEPTIONS — Do NOT Replace

- **Intentional LTR blocks**: Elements with `ltr` class or `dir="ltr"` — numbers, currency, percentages are often intentionally LTR. In this codebase, `ltr text-right` is a common pattern for displaying numbers right-aligned in LTR direction. Leave these alone.
- **Dialog/modal centering**: `left-1/2 -translate-x-1/2` for centering is direction-independent. Leave alone.
- **Chart library margins**: Recharts/chart library config objects like `margin={{ left: 0, right: 0 }}` are not CSS — leave alone.
- **Toggle/switch positioning**: `left-[3px] right-auto` patterns for toggle knob position are physical by design. Leave alone.
- **Progress bar absolute positioning**: `left: X%` with `style=` for progress bar fills — these are math-based, not directional.
- **shadcn/ui components**: Files in `src/components/ui/` that are generated by shadcn — only fix if clearly broken, as these have their own conventions.

## CATEGORY 2: Directional Icons

Icons that indicate direction must be flipped in RTL contexts.

### Icons That MUST Flip

- Arrow left/right (navigation: back, forward, next, previous)
- Chevron left/right (breadcrumbs, carousels, pagination)
- External link arrow
- Reply / forward icons
- Undo / redo icons
- Indent / outdent icons
- List bullet alignment indicators
- Logout/login door icons with arrows

### Icons That Must NOT Flip

- Search (magnifying glass)
- Checkmarks
- X / close
- Plus / minus
- Media controls (play, pause, stop) — these represent tape direction, not text direction
- Warning triangles
- Settings / gear
- User / avatar
- Download / upload arrows (vertical)
- Refresh / sync (circular)
- Home
- Star / heart / flag

### How to Flip

```tsx
// Method 1: Tailwind rtl variant
<ChevronRight className="rtl:-scale-x-100" />

// Method 2: CSS custom property
// In global CSS:
[dir="rtl"] { --icon-dir: -1; }
[dir="ltr"] { --icon-dir: 1; }
// On icon:
<Icon style={{ transform: `scaleX(var(--icon-dir))` }} />

// Method 3: Swap icon entirely
{isRTL ? <ChevronLeft /> : <ChevronRight />}
```

### Detection

Search for: `ChevronRight`, `ChevronLeft`, `ArrowRight`, `ArrowLeft`, `ArrowUpRight`, `ExternalLink`, `Reply`, `Forward`, `Undo`, `Redo` from lucide-react. Check if they have RTL flip handling.

## CATEGORY 3: Box Shadow and Gradients

### Box Shadow

Horizontal offset in `box-shadow` does NOT auto-flip in RTL. A shadow on the right in LTR should be on the left in RTL.

**Detection**: Search for `box-shadow` with non-zero horizontal offset, or Tailwind `shadow-*` with custom values.

**Fix**: Use CSS custom property:
```css
[dir="ltr"] { --shadow-dir: 1; }
[dir="rtl"] { --shadow-dir: -1; }
.card { box-shadow: calc(4px * var(--shadow-dir)) 4px 8px rgba(0,0,0,0.2); }
```

### Background Gradients

`background: linear-gradient(to right, ...)` does NOT auto-flip.

**Fix**: Use logical keywords:
```css
/* WRONG */
background: linear-gradient(to right, red, blue);
/* CORRECT */
background: linear-gradient(to inline-end, red, blue);
/* Or use Tailwind */
className="bg-gradient-to-r rtl:bg-gradient-to-l"
```

### Background Position

`background-position: left center` does NOT auto-flip.

**Fix**: Use logical values or Tailwind `bg-start` / `bg-end` if available.

## CATEGORY 4: Transforms and Animations

### translateX Does NOT Auto-Flip

`transform: translateX(-100%)` will move in the same physical direction regardless of `dir`. This breaks slide-in/out animations.

**Detection**: Search for `translateX`, `-translate-x-`, `translate-x-` in animation contexts.

**Fix**:
```css
[dir="ltr"] { --dir: 1; }
[dir="rtl"] { --dir: -1; }
.slide-in { transform: translateX(calc(-100% * var(--dir))); }
```

Or in Tailwind:
```html
<div className="translate-x-full rtl:-translate-x-full">
```

### CSS Animations with Directional Movement

`@keyframes` with `translateX` or `left`/`right` values need RTL variants.

**Detection**: Search for `@keyframes` containing `translateX` or horizontal movement.

## CATEGORY 5: Bidirectional Text Issues (Hebrew-Specific)

### Numbers in Hebrew Text

Numbers are always LTR, even in RTL text. The Unicode Bidi Algorithm handles this automatically in most cases, but issues arise with:

- **Phone numbers with dashes**: `054-123-4567` may render as `054-123-4567` or get garbled. Wrap in `<span dir="ltr">` or `<bdi>`.
- **Negative numbers**: `-500` — the minus sign may appear on wrong side. Use `<bdi>` or `dir="ltr"`.
- **Currency with numbers**: `₪1,500` — the shekel sign may jump around. Best practice: wrap the entire value in `<span dir="ltr">₪1,500</span>`.

### Parentheses and Brackets

Parentheses are "mirrored" characters in Unicode — `(` becomes `)` visually in RTL context. This works automatically but breaks when mixing Hebrew and English:
- `הוצאה (Expense)` — may render with parentheses on wrong sides
- Fix: Use `<bdi>` around embedded LTR text: `הוצאה <bdi>(Expense)</bdi>`

### Punctuation at Text Boundaries

Period, comma, exclamation, question mark at the start/end of mixed-direction text can jump to wrong position.
- **Detection**: Look for string interpolation mixing Hebrew and numbers/English, especially at line boundaries.
- **Fix**: Use `<bdi>` or explicit `dir="ltr"` on the embedded content.

### Common Hebrew Patterns

```tsx
// WRONG — period may jump
<span>סה"כ: {total}</span>

// CORRECT — isolate the number
<span>סה"כ: <bdi>{formatCurrency(total)}</bdi></span>

// WRONG — parentheses may mirror incorrectly
<span>{name} ({email})</span>

// CORRECT
<span>{name} <bdi>({email})</bdi></span>
```

### The `ltr` Class Pattern

In this codebase, the pattern `ltr text-right` is used on number-displaying elements. This forces LTR direction for the number while right-aligning it (so it appears on the correct side in the RTL layout). This is intentional — do not remove.

However, verify:
1. All currency/number displays use this pattern consistently
2. The `ltr` class actually sets `direction: ltr` (check globals.css)

## CATEGORY 6: Form Inputs

### Email and URL Inputs

Email and URL inputs should always be LTR, even in RTL interfaces.

**Fix**:
```css
input[type="email"],
input[type="url"],
input[type="tel"] {
  direction: ltr;
  text-align: right; /* keeps right-aligned in RTL layout */
}

/* Show placeholder on correct side */
input[type="email"]:placeholder-shown {
  direction: ltr;
}
```

### Number Inputs

Number inputs should display LTR (digits flow left-to-right).

**Detection**: Search for `<input type="number"` or number input fields without `dir="ltr"` or `ltr` class.

### Placeholder Direction

Placeholders in RTL inputs should match the expected input language:
- Hebrew placeholder in Hebrew input: RTL (default, works fine)
- English/number placeholder: may need `dir="ltr"` or `:placeholder-shown` styling

### Search Input

If search accepts both Hebrew and Latin, use `dir="auto"`:
```html
<input type="search" dir="auto" placeholder="חיפוש..." />
```

## CATEGORY 7: Scrollbar and Overflow

### Horizontal Scrollbar Position

In RTL, horizontal overflow goes to the LEFT (opposite of LTR). Some browsers handle this inconsistently.

**Known Issue**: Firefox has historical bugs with horizontal scrollbars on RTL pages.

**Detection**: Look for `overflow-x: auto/scroll` on containers in RTL context.

**Fix**: Test manually. Ensure `100vw` doesn't cause phantom horizontal scroll:
```css
/* Prevent 100vw scrollbar issue in RTL */
html[dir="rtl"] {
  overflow-x: hidden; /* if no horizontal scroll is needed */
}
```

### Vertical Scrollbar Position

In RTL, the vertical scrollbar should appear on the LEFT side. Most browsers handle this correctly, but custom scrollbar styling may break it.

**Detection**: Check custom scrollbar CSS (`::-webkit-scrollbar`) for left/right positioning assumptions.

## CATEGORY 8: Flexbox and Grid Specifics

### Flexbox — Generally RTL-Safe

Flexbox `row` direction auto-reverses in RTL. `justify-content: flex-start` maps to the right in RTL. No action needed for basic flex layouts.

**Exception**: `space-x-*` in Tailwind uses `margin-left` under the hood. In RTL, you may need `space-x-reverse`:
```html
<div className="flex space-x-4 space-x-reverse">
```
Or better, use `gap-*` which is direction-independent:
```html
<div className="flex gap-4">
```

**Detection**: Search for `space-x-` without accompanying `space-x-reverse` or `rtl:space-x-reverse`. Suggest replacing with `gap-*`.

### Grid — Generally RTL-Safe

CSS Grid auto-reverses column flow in RTL. No special handling needed for basic grids.

**Exception**: Named grid areas with directional names (e.g., `grid-area: left-panel`) are semantically confusing but functionally fine.

### Table Column Order

Tables auto-reverse column order in RTL. The rightmost `<td>` in HTML becomes the first visual column. This is correct behavior.

## CATEGORY 9: Third-Party Libraries

### Recharts / Chart Libraries

Chart libraries typically don't respect RTL. Charts remain LTR (which is usually correct for data visualization).

**Check**: Ensure chart containers have `dir="ltr"` if the chart looks broken, or that the chart wrapper uses the `ltr` class.

### Date Pickers

Date/calendar components may not handle RTL well. Verify:
- Calendar reads right-to-left (Sunday on far right for Hebrew locale)
- Navigation arrows are correct direction
- Month/year text is properly aligned

### Toast/Notification Libraries (Sonner, etc.)

Toast notifications may slide in from wrong side. Check animation direction.

## CATEGORY 10: CSS `margin-right` for Sidebar Layout

In this specific codebase, the sidebar layout uses `margin-right: var(--sidebar-width)` in `globals.css`. This is correct for RTL — the sidebar is on the right, so content needs right margin. However, if the app ever needs LTR support, this should become `margin-inline-end`.

## SCANNING PROCEDURE

When running this skill, follow this order:

1. **Scan globals.css** for physical properties in CSS rules
2. **Scan all .tsx files** for physical Tailwind classes (ml-, mr-, pl-, pr-, left-, right-, text-left, text-right, border-l-, border-r-, rounded-l-, rounded-r-)
3. **For each match**, check if it's in an exception case (ltr block, chart config, shadcn component, dialog centering, toggle)
4. **Check icon usage** — search for directional Lucide icons and verify RTL flip handling
5. **Check form inputs** — verify email/url/tel/number inputs have proper direction
6. **Check currency/number displays** — verify consistent use of `ltr` class + `<bdi>` where needed
7. **Check `space-x-*`** usage — suggest `gap-*` replacement
8. **Check transforms/animations** — verify translateX values have RTL handling
9. **Report findings** in a structured format with file:line and recommended fix

## OUTPUT FORMAT

For each issue found, report:
```
FILE: src/app/example/page.tsx:42
ISSUE: Physical class `mr-2` used instead of logical `me-2`
CONTEXT: <span className="mr-2 text-sm">
FIX: Replace `mr-2` with `me-2`
SEVERITY: low|medium|high
```

Severity guide:
- **high**: Layout is visually broken in RTL (wrong side positioning, reversed reading order)
- **medium**: Inconsistent with RTL best practices, may cause subtle visual issues
- **low**: Technically a physical property but in a context where it doesn't matter (symmetric, centering, etc.)
